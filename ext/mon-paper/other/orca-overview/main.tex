\section{ORCA: Overview}\label{orca-overview}

ORCA (\emph{Observability with Realtime Control and Aggregation}) is a programmable, low-overhead observability framework for \emph{bulk-synchronous parallel} (BSP) applications, designed to run asynchronously alongside production MPI workloads with minimal and optional integration effort. It enables continuous, interactive observability by treating telemetry as a structured dataflow, windowed by timestep and processed in real time using a distributed SQL-based engine backed by Apache Arrow~\cite{:ApacheArrow} and Parquet~\cite{:ApacheParquet}.
ORCA is built to run alongside large scale simulations (O(10k) nodes or more), leveraging high performance userspace fabrics and distributed query planning techniques such as operator pushdown to minimize data movement for a given observability task.
Unlike tracing or postmortem profiling tools, ORCA is designed to support the full spectrum of observability workflows, from lightweight common-case monitoring to root cause localization via realtime and dynamic telemetry streams.
A dedicated control path provides correctness guarantees for interactions such as toggling probes or modifying aggregation logic, using a custom asynchronous protocol designed to maintain consistent control semantics.
ORCA aims to improve large cluster efficiency by rapidly closing the loop between detection, localization, and mitigation of performance artifacts. A logical view of ORCA is provided in Fig. \ref{fig:logical}.

\section{Motivation}
\subsection{Lessons from Optimizing Parallel Codes}

ORCA emerged from two years of experience optimizing parallel applications at
scale---up to 4,096 cores on an in-house cluster. While load imbalance is
typically viewed as the primary source of inefficiency in such codes, we
consistently encountered other performance bottlenecks: fail-slow hardware,
transient network latency spikes, and suboptimal execution strategies, among
others (see Fig.~\ref{fig:vol}). These sources of variability are pervasive,
often transient, and highly application-specific, requiring case-by-case
mitigation. An example of network-level variability observed on our cluster with a real application is shown in Fig. ~\ref{fig:vol}. While this variability originated on a few ranks and affected the entire application, it was not visible in aggregate telemetry. A localization exercise taking us two months revealed the source to be a network driver tuning issue, which was mitigated in a few days once localized. Because BSP applicaitons are acutely sensitive to such disruptions, effective mitigation requires costant and real-time visibility into live application behavior.

Existing tooling falls short of these visibility requirements along multiple
axes. Profiling tools~\cite{Shend2006:TAU} offer only fixed, coarse-grained aggregates---often
averages over long execution windows---which obscure transient outliers and
fail to capture the structure of BSP synchronization patterns. Tracing~\cite{Shend2006:TAU,Wylie2025:15Years}
records detailed event streams, but the resulting data is often rigidly
structured, hard to query, and poorly suited for isolating noisy, emergent
patterns across time and ranks. Worse, both profiling and tracing are
inherently postmortem: they require the application to terminate before any
meaningful analysis can begin. Sampling tools, such as LDMS~\cite{Schwa2021:LDMS}, do support
real-time telemetry and are adequate for basic cluster-level monitoring, but
they collect uncoordinated snapshots that discard the spatial and temporal
structure of the problem. Additionally, all these tools assume fixed telemetry
schemas and analysis workflows, whereas we found a pressing need for dynamic,
interactive, and domain-specific observability pipelines that evolve with the
problem. Our early attempts to bridge these gaps evolved into an ad hoc OLAP pipeline---
ORCA is a principled generalization of that design.

\subsection{Design Goals for Ideal Observability}

Our experience led us to identify three core requirements for an ideal
observability framework in HPC:

\begin{itemize}
	\item \textbf{Programmability.} Users should be able to specify what data
	      sources to tap, what schemas to expose, and how to aggregate the resulting
	      telemetry. This enables domain-aware customization of the observability
	      pipeline while maintaining a common execution substrate. Examples include combining performance telemetry with application-level context (\texttt{rank ID}, \texttt{timestep} etc.), programmable aggregation (averages, approximate quintiles, or all data), and programmable triggers/thresholds for data collection.

	\item \textbf{Efficient real-time analytics.} Insights must be produced as
	      the application runs, with minimal delay and overhead. The system should
	      preserve the structured nature of BSP execution---especially timestep
	      alignment---to make telemetry interpretable in the context of the program’s
	      control flow.

	\item \textbf{Interactivity.} Effective observability must support a full
	      spectrum of workflows, from low-overhead common-case monitoring to detailed
	      root-cause localization. This requires the ability to dynamically insert
	      probes, reconfigure aggregations, and adapt queries mid-execution, all via a
	      control path that maintains consistency with application progress.
\end{itemize}

\section{ORCA: Design Overview}

ORCA is organized as a modular, declarative telemetry engine built atop a
high-performance transport overlay. This section describes how data and control
flow through the system---from instrumentation to aggregation to control---and
how the system is designed to minimize overhead while maximizing flexibility
and correctness.

\paragraph{Overlay and bootstrapping.} ORCA bootstraps a tree-based overlay
alongside the MPI application. This overlay consists of a controller
and a tier of aggregators, each of which is assigned a unique overlay identifier
(\texttt{ovid}) and communicates using Mercury~\cite{Souma2013:MercuryRPC}, a high-performance RPC library
for HPC fabrics. MPI ranks are mapped to aggregators, forming
a reduction tree for telemetry and a broadcast tree for control. Bootstrap
coordination occurs via a lightweight TCP protocol, after which all
communication proceeds over the high-performance fabric. ORCA is amenable to
its communication being assigned a lower network priority than the application.

\paragraph{Telemetry collection.} Each MPI rank can instantiate one or more
data collectors, which emit telemetry conforming to Arrow-compatible schemas.
Collectors are schema-specific and encapsulate probes for a given API domain
(e.g., MPI, Kokkos, NCCL). Events are collected as rows in-memory, and collectors
are drained once every timestep, towards the end, to minimize RPC count and jitter.
For schemas with the column \texttt{probe\_name}, ORCA automatically announces probe
names to the controller, and data for that probe is collected only if it is enabled.
Collection hooks devolve to \emph{noops} for inactive probes.
% Collectors are flushed exactly once at the end of
% each timestep, minimizing RPC count and jitter. Probes can be individually
% toggled via control commands; when inactive, their collection paths devolve to
% no-ops. 
Adding new telemetry streams requires no changes to core
infrastructure---just define a new collector with the desired schema.

\paragraph{Aggregation.} Collected telemetry is grouped by timestep and routed
through the overlay for in-situ aggregation. ORCA uses Apache Arrow and
Datafusion~\cite{Lamb2024:DataFusion} to implement in-memory SQL-based aggregation with distributed query
optimizations such as projection, filtering, and partial aggregation pushdown, as close to source as possible.

\paragraph{Sinks.} Aggregated results are routed to \emph{sinks} --- our abstraction for handling data exiting our dataflow pipeline. Sinks can capture data either at the controller, or directly at aggregators. Our current sinks include a DuckDB~\cite{Raasv2019:DuckDB} instance at the controller to serve a live dashboard, and a Parquet sink used to persist partitioned high-volume telemetry to a parallel filesystem directly from the aggregation tier. Additional sinks can be configured as storage/indexing requirements evolve.

\paragraph{Control path.} ORCA’s control messages are disseminated using a
custom two-phase commit protocol aligned to application timesteps, called \tstpc.
\tstpc{} leverages the fact that the controller has a lagging view of the application
timestep at any given point, and asynchronously launches a control command for a future
timestep ($t + \delta$). Commands are committed if received by all ranks before $t + \delta$,
and aborted otherwise --- ranks never block, and correctness is maintained either way. $\delta$ is a function of
the configured parameters and is learned automatically by the control path.

\section{Usability}

\paragraph{FlightSQL and Grafana integration.} ORCA implements a DuckDB~\cite{Raasv2019:DuckDB}-based
telemetry sink at the controller, wrapped around an inbuilt FlightSQL~\cite{:ArrowFlightSQL}
interface. This enables users to query live telemetry data via SQL, directly
from tools such as Grafana~\cite{:GrafanaDashboard}. The integration provides a low-friction way to
visualize application behavior using conventional dashboards. In addition, an
experimental Arrow Flight~\cite{:ArrowFlight} server offers lower-latency pub/sub access for lower-latency streaming telemetry.

\paragraph{The OrcaFlow DSL.} ORCA provides a declarative telemetry DSL called
\texttt{OrcaFlow}, designed to systematize telemetry workflows as structured
dataflows. Each flow defines a set of SQL transformations and targets, allowing
users to filter, aggregate, and process telemetry in a reproducible way. Flows
can be activated or retired dynamically, supporting interactive
performance debugging, anomaly detection, and long-running telemetry
collection---all within a unified dataflow model.

\paragraph{Integration requirements.} ORCA can collect basic telemetry and provide
aggregation and control semantics without application modification, if used via a preloaded
plugin. Richer observability capabilities, however, require applications to expose their timestep
by linking with ORCA and invoking a hook. We expect applications linking with ORCA to not
require the service to be active at runtime for the application to execute correctly.

\section{ORCA as a Substrate: Use Cases and Open Directions}

While ORCA initially evolved in response to performance challenges we encountered with
large-scale scientific codes, we quickly realized that its core abstractions---
structured telemetry, declarative aggregation, and timestep-aligned control---
enable a broader class of introspection and control workflows within
bulk-synchronous systems. Beyond performance monitoring, ORCA allows synthesized
views of arbitrary system layers to be surfaced at runtime---such as driver queues, asynchronous pipelines, internal application state---without interrupting the
application.

These views are computed through dataflows optimized using pipelined,
distributed query execution techniques, and can be adapted on the fly. Coupled
with a consistent control path, ORCA completes the feedback loop: users can
observe, analyze, and act on runtime behavior from within the same framework,
without intrusive instrumentation or coordination barriers. 
These workflows are built dynamically, updated live, and can evolve over time by accumulating libraries of operators, placement strategies, and dataflow source/sink abstractions.

We describe below several directions we are currently exploring. These are
intended to convey the range of capabilities ORCA enables, and to help identify
whether similar questions or gaps arise in other domains.

\begin{itemize}

\item \emph{Programmable aggregation and selective persistence}:
ORCA supports real-time aggregation at variable resolutions. Users can monitor
low-resolution summaries by default and selectively enable fine-grained telemetry
during anomalous behavior or debugging. Telemetry can be filtered and
conditionally persisted as Parquet, partitioned by timestep and rank, enabling
efficient post hoc analysis without overwhelming storage.

\item \emph{Dynamic probing via control path}:
ORCA's control plane allows probes to be activated or deactivated at runtime,
enabling fine-grained observability to be deployed only when and where needed.
Beyond preload wrappers, ORCA supports probe insertion via tools like eBPF~\cite{:eBPF} and
DynInst~\cite{Berna2011:DynInst}. Combined with programmable aggregation, this supports seamless runtime
transitions from lightweight monitoring to detailed root cause localization,
without restarting or perturbing the application.

\item \emph{Operator-in-the-loop tuning}:
ORCA enables interactive tuning as a natural consequence of its unified control
and aggregation model. For runtime parameters exposed via the framework, ORCA
can guarantee consistent updates across all ranks and apply them at known
timestep boundaries. This allows for safe mid-execution adjustments---for
example, to load-balancing thresholds or sampling rates---driven by real-time
telemetry.

\item \emph{Live introspection of application state}:
ORCA allows applications to expose internal state (e.g., particle energies,
residuals, or mesh quality metrics) as queryable streams. Dataflows can be
augmented with lossy aggregations such as sketches, enabling high-frequency
summaries with reduced overhead. This makes it feasible to monitor rich
scientific state without resorting to large-scale checkpoints.

\item \emph{Future direction---leveraging accelerators}:
ORCA's dataflow model can be extended to incorporate SmartNICs, GPUs, or
other specialized hardware as compute nodes within the telemetry pipeline.
By treating these devices as executor targets for query fragments, ORCA opens
the door to fine-grained in-network or near-data processing.

\end{itemize}

We believe ORCA is directly relevant to machine learning workloads, both training and inference, because of their shared scale and bulk-synchronous characteristics.
While performance telemetry is an obvious use case---including
analysis of collective stragglers, pipeline stalls, or asynchronous data loading---
ORCA also supports introspection of application state, such as monitoring weight
distributions or gradient sparsity during training. Given an arbitrary view of
cluster state defined as a synthesis of telemetry, ORCA can orchestrate the corresponding dataflow to compute and act on it.
We are interested in whether these capabilities fill a meaningful gap relative to
your current tooling---both in terms of performance visibility and application-level
state introspection---and would appreciate your perspective on where ORCA’s
model could be useful or complementary.
